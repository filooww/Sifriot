# Story 1.25: Category to Section Refactoring

## Status
Approved

## Story

**As a** system administrator and site maintainer,
**I want** the hierarchical Category entity renamed and re-scoped to Section across the entire system (database, backend, API/URL contracts, frontend UI, and documentation),
**so that** the taxonomy is correctly understood as a pure navigational/organizational structure (where content lives in the site) rather than a semantic content classifier (what content is about), eliminating overlap with Genres and Content Types.

## Acceptance Criteria

### 1. Database Schema Rename

1. Create a new migration that renames the `categories` table to `sections`
2. Create a new migration that renames the `category_publication` pivot table to `section_publication`
3. In the `section_publication` pivot table, rename column `category_id` to `section_id`
4. In the `sections` table, the self-referencing `parent_id` foreign key constraint must be updated to reference `sections.id`
5. All indexes on the renamed tables must be recreated with correct names
6. The migration must be reversible (`down()` method restores original names)
7. Existing data in both tables must be preserved (rename only, no data loss)

## Dev Notes

### Conceptual Change
This is a **rename + reseed**, not a structural change. The hierarchical self-referencing pattern, many-to-many relationship with publications, multilingual support, soft deletes, and sort ordering all remain identical. The key change is:
- **Before**: Category = content classifier ("Fiction", "Science", "Research Papers") -- overlaps with Genres and Content Types
- **After**: Section = navigation structure ("Featured", "New Arrivals", "Browse by Topic") -- orthogonal to Genres and Content Types

### Database Migration Strategy
[Source: docs/architecture/coding-standards.md]

Use Laravel's `Schema::rename()` for table renames and `$table->renameColumn()` for column renames. These are safe operations that preserve data. The migration must handle:

1. Drop existing FK constraints before renaming (MySQL requires this)
2. Rename tables
3. Rename columns
4. Recreate FK constraints and indexes with new names
5. `down()` reverses all of these

**Critical**: The `publications` table uses non-standard primary key `id_publication`. The pivot table FK `publication_id` references `publications.id_publication`. This custom FK mapping must be preserved.

Example migration pattern:
```php
// Up
Schema::table('category_publication', function (Blueprint $table) {
    $table->dropForeign(['category_id']);
    $table->dropForeign(['publication_id']);
    $table->dropUnique(['category_id', 'publication_id']);
});
Schema::rename('category_publication', 'section_publication');
Schema::table('section_publication', function (Blueprint $table) {
    $table->renameColumn('category_id', 'section_id');
});
// Then recreate constraints with new names
```

### Current Category Model Structure
[Source: app/Models/Category.php]

```php
class Category extends Model {
    use HasFactory, SoftDeletes;
    protected $fillable = ['parent_id', 'name_en', 'name_ru', 'name_he', 'slug', 'sort_order'];
    protected $casts = ['sort_order' => 'integer'];

    public function parent(): BelongsTo // self-ref
    public function children(): HasMany // ordered by sort_order
    public function publications(): BelongsToMany // via category_publication
    public function getLocalizedNameAttribute(): string // locale-aware
}
```

This structure is preserved exactly -- only class/table/FK names change.
