# Story 1.7: Bulk Folder Scanning and Cataloging

## Status
Ready for Review

## Story

**As an** administrator,
**I want** to scan entire folders of existing files and catalog them in bulk,
**so that** I can efficiently register the entire 1.1TB library.

## Acceptance Criteria

1. Bulk scan interface allows admin to select folder path from configured external library paths or internal storage via embedded folder tree navigation
1a. Bulk scan page includes embedded folder tree navigation component (Windows Explorer-style sidebar)
1b. Folder tree displays all admin-configured external library paths as root nodes
1c. Admin can expand/collapse folders to navigate tree structure
1d. Selected folder path automatically populates scan path field
2. Admin can configure scan options: recursive (include subfolders), file format filters, max depth
3. Scan runs as background queue job (async processing)
4. System recursively processes all files in selected folder and subfolders
5. External server folder structure preserved as organizational metadata (full external path stored in `original_folder_path`)
6. Each discovered file processed as separate queue job
7. Bulk scan dashboard shows real-time progress: "Processing: 450/2,847 files | Success: 420 | Skipped: 25 | Failed: 5"
8. Already-registered files automatically skipped (no duplicates)
9. Failed registrations listed with error reasons (invalid format, size exceeded, read permission denied)
10. Successful scans create draft publication records with "Pending" status
11. Bulk scan summary report generated after completion (total found, registered, skipped, failed, processing time)
12. Admin can pause/cancel bulk scan in progress
13. Scan results filterable (show only errors, show only new registrations)
14. Admin can bulk-approve scanned publications or review individually

## Integration Verification

- **IV1**: Queue worker processes scan jobs successfully (`php artisan queue:work`)
- **IV2**: Failed jobs stored in `failed_jobs` table with full context for debugging
- **IV3**: Large bulk scans (10,000+ files) don't crash, timeout, or cause memory issues
- **IV4**: Scan progress updates visible in real-time via Livewire
- **IV5**: Canceling scan stops new jobs and doesn't corrupt existing data

### Tech Stack Context
[Source: [tech-stack.md](../architecture/tech-stack.md)]

**Backend Framework**: Laravel 12
- Use Eloquent ORM for all database interactions
- Leverage Laravel Storage facade for file operations
- Queue system: Laravel Queue (database driver) for background jobs

**Queue System**: Laravel Queue (database driver)
- Jobs implement `ShouldQueue` interface
- Configure retry attempts: `public int $tries = 3`
- Configure timeout: `public int $timeout = 600` (10 minutes for discovery, 2 minutes for registration)
- Failed jobs automatically stored in `failed_jobs` table

**Interactive UI**: Livewire 3.6.4
- All admin components built with Livewire
- Use `wire:model` for form bindings
- Use `wire:poll.2s="refreshProgress"` for real-time progress updates
- Dispatch events for component communication

**File Operations**: Laravel Storage + league/flysystem 3.0
- Use `Storage` facade for all file operations
- `Storage::disk('local')->allFiles($path)` for recursive file listing
- `Storage::disk('local')->files($path)` for non-recursive listing
- `Storage::disk('local')->exists($path)` for file existence checks

**Logging**: Laravel Logging + Custom ColoredLineFormatter
- Custom log channel `folder_scan` for bulk scan operations
- ColoredLineFormatter with ANSI codes: Green (info), Yellow (warning), Red (error)
- Logs stored in `storage/logs/folder-scan.log`

**Database**: MySQL 8.0
- InnoDB engine for transactions
- UTF8mb4 character set for multi-language support

**Testing**: PHPUnit 11.5.3
- Feature tests for HTTP flows and Livewire components
- Unit tests for services and models
- Test database: `literature_test` (configured in phpunit.xml)

- Columns: `id`, `user_id`, `folder_path`, `scan_options` (JSON), `status` (enum: pending/processing/paused/completed/cancelled/failed), `total_files_found`, `files_registered`, `files_skipped`, `files_failed`, `processing_time_seconds`, `started_at`, `completed_at`, `created_at`, `updated_at`
- Relationships: `belongsTo(User)`, `hasMany(FileRegistrationLog)`
- Accessor: `getProgressPercentAttribute()` calculates progress percentage

**FileRegistrationLog Model** (from Story 1.6):
- Extend to support bulk scan: `folder_scan_job_id` column links to `FolderScanJob`
- `registration_source` value: 'bulk_scan' for bulk-scanned files
- `status` enum: 'pending', 'processed', 'failed'
- `error_message` stores failure reasons

**Publication Model** (extend existing):
- Add column: `original_folder_path` (string, nullable, index)
- Stores original filesystem path for organizational metadata
- Helps admins identify file location for troubleshooting

**Queue Jobs**:
- `DiscoverFilesJob`: Discovers all files in folder, dispatches registration jobs
- `ProcessFileRegistrationJob`: Registers individual file, creates Publication and File records
- Both implement `ShouldQueue` interface
- Failed jobs stored in `failed_jobs` table with full context
