# Story 1.24: Gemini LLM Metadata Extraction

## Status
Ready for Review

## Story

**As an** administrator reviewing publication metadata,
**I want** an "Extract with AI" button that sends document content to Google Gemini LLM for intelligent metadata extraction,
**So that** I can quickly populate metadata fields with high-accuracy AI-extracted data when traditional extraction methods fail or return incomplete results.

## Acceptance Criteria

### 1. Gemini API Integration Service

1. Create `GeminiMetadataExtractorService` in `app/Services/MetadataExtractors/`
2. Service implements `MetadataExtractorInterface` for consistency with existing extractors
3. Service uses Google Gemini API (gemini-1.5-flash or gemini-1.5-pro model)
4. API key configured via environment variable `GEMINI_API_KEY`
5. Add `config/services.php` entry for Gemini configuration (api_key, model, max_tokens)
6. Service accepts extracted text content (not raw file) as input
7. Service returns `ExtractedMetadata` DTO with confidence scores based on Gemini's response

### 2. Text Content Preparation

8. Create `DocumentTextExtractor` helper service to extract first N pages of text from documents
9. Support extraction from: PDF (smalot/pdfparser), EPUB, DOCX (phpoffice/phpword), FB2, TXT
10. Configurable page/character limit via config (default: first 5000 characters or ~5 pages)
11. Text is cleaned, normalized to UTF-8, and truncated to fit Gemini token limits
12. Helper reuses existing extractor logic where possible (PDFMetadataExtractor, etc.)

### 3. Gemini Prompt Engineering

13. Create structured prompt that requests JSON response with specific fields
14. Prompt requests extraction of: title, authors (array), publication_year, publisher, genres (array)
15. Prompt includes instructions for confidence scoring (0.0-1.0) for each field
16. Prompt handles multilingual content (English, Russian, Hebrew)
17. Response parsing validates JSON structure and handles malformed responses gracefully
18. Fallback to empty `ExtractedMetadata` if Gemini returns invalid data

### 4. MetadataReviewForm Integration

19. Add "Extract with AI" button to `MetadataReviewForm` component (visible when file exists)
20. Button shows loading state with spinner during API call
21. Button disabled if `GEMINI_API_KEY` not configured (show tooltip explaining why)
22. On click: extract text from publication's primary file, send to Gemini service
23. On success: populate form fields with extracted data as text values, show success notification
24. On error: show error notification with message, log detailed error to `folder_scan` channel
25. Extracted data includes confidence badges (color-coded) for each field
26. User can accept/modify AI-extracted values before confirming
27. AI-extracted authors/genres populate as text strings in the existing input fields
28. Admin uses existing autocomplete to match AI suggestions to existing entries OR create new ones
29. No automatic matching logic - admin retains full control over taxonomy decisions

### 5. Error Handling and Rate Limiting

30. Implement retry logic with exponential backoff (max 3 retries)
31. Handle Gemini API errors: rate limit (429), quota exceeded, invalid API key, timeout
32. Log all Gemini API calls with request/response details to `folder_scan` log channel
33. Track API usage metadata in `file_metadatas` table (`extraction_method = 'gemini_llm'`)
34. Configurable timeout for API calls (default: 30 seconds)

### 6. Configuration and Environment

35. Add to `.env.example`: `GEMINI_API_KEY=`, `GEMINI_MODEL=gemini-1.5-flash`, `GEMINI_MAX_CHARS=5000`
36. Add config validation that warns if API key is missing (not hard fail)
37. Document API key setup in README or inline help text

### 7. Security Considerations

38. API key never exposed to frontend (all calls server-side only)
39. Sanitize document text before sending (remove potentially sensitive paths/metadata)
40. Rate limit extraction requests per user session (prevent abuse)
41. Admin-only access to AI extraction feature (existing auth check sufficient)

## Dev Notes

### Existing Metadata Extraction Architecture
[Source: app/Services/MetadataExtractors/]

The project already has a robust metadata extraction system:
- `MetadataExtractorInterface` defines `extract(string $filePath): ExtractedMetadata`
- `AbstractMetadataExtractor` provides shared utilities (encoding, year extraction, logging)
- `ExtractedMetadata` DTO stores fields with confidence scores (0.0-1.0)
- Individual extractors exist for: PDF, DOCX, EPUB, FB2, TXT, DOC, DJVU
- `MetadataExtractorFactory` selects extractor by file extension

**Key Insight**: The Gemini extractor should NOT implement `MetadataExtractorInterface` directly because it doesn't take a file path - it takes pre-extracted text. Create a separate service that composes with existing extractors.

### Gemini API Reference

**Endpoint**: `https://generativelanguage.googleapis.com/v1beta/models/{model}:generateContent`

**Request format**:
```json
{
  "contents": [{
    "parts": [{"text": "your prompt here"}]
  }],
  "generationConfig": {
    "temperature": 0.1,
    "maxOutputTokens": 1024
  }
}
```

**Authentication**: API key in header `x-goog-api-key: {key}` or query param `?key={key}`

**Recommended model**: `gemini-1.5-flash` (fast, cost-effective) or `gemini-1.5-pro` (higher quality)

### MetadataReviewForm Context
[Source: app/Livewire/Admin/MetadataReviewForm.php]

The form already handles:
- Loading metadata from `FileMetadata` model
- Confidence scores display with color coding
- Multiple extraction methods tracking
- Success/error notifications via `$this->dispatch('notify', ...)`
- Author/genre/publisher autocomplete with creation

The AI extraction should integrate seamlessly with existing flow:
1. User opens metadata review modal
2. User clicks "Extract with AI" button
3. Fields populate with AI-extracted data as **raw text values** (confidence shown)
4. User reviews AI suggestions:
   - For authors/genres/publishers: AI text appears in input fields
   - User types in autocomplete → existing matches shown → select existing OR create new
   - **No automatic matching** - admin decides whether AI suggestion matches existing entry
5. User confirms extraction (existing flow)

**Design Decision**: Gemini extracts raw values without awareness of existing taxonomy. This keeps the system flexible and gives admins full control. The existing autocomplete UI already handles matching/creation elegantly.

### FileMetadata Model
[Source: app/Models/FileMetadata.php]

Key fields:
- `extraction_method`: string (e.g., 'pdf_parser', 'manual_entry', 'gemini_llm')
- `extracted_data`: JSON with structure `{field: {value: ..., confidence: 0.x}}`
- `confidence_scores`: JSON with overall field confidences
- Status flow: pending → processed → confirmed/rejected
