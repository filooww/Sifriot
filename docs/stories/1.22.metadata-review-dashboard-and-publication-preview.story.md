# Story 1.22: Metadata Review Dashboard and Publication Preview Enhancement

## Status
Approved

## Story

**As an** administrator,
**I want** a dedicated metadata review page at `/dashboard` with autocomplete-enhanced metadata fields and publication status controls, plus a public publication preview page for guests,
**so that** I can efficiently manage publication metadata as my primary workspace and provide engaging preview experiences for visitors before they authenticate.

## Acceptance Criteria

### 1. Dashboard Restructuring (Metadata Review as Primary Workspace)

1. The Metadata Review Queue (currently a tab) shall be moved to become the primary content of `/dashboard`
2. The existing AdminDashboard publication list functionality shall be relocated/refactored to support the metadata review workflow
3. Admin dashboard shall display:
   - Metadata review queue list with filters (status: pending/processed/confirmed/failed/rejected)
   - Quick stats cards: Total pending, Processed awaiting review, Confirmed, Failed
   - Inline metadata review form (expandable per item) or modal-based review
4. Dashboard filtering shall use the existing PublicationFilters component architecture from AdminDashboard
5. Search functionality shall remain on the dashboard page (not in navigation)

### 2. Publication Status Toggle (Publish/Unpublish)

6. Publications shall have a publish/unpublish toggle visible on:
   - Metadata review dashboard (each publication row)
   - Publication detail admin view
   - Bulk actions (select multiple → publish/unpublish all)
7. Status toggle shall update the `publications.status` field (enum: 'published', 'hidden', 'pending')
8. Published publications appear in public search/browse results
9. Hidden/Pending publications invisible to guests and regular users
10. Status changes shall be logged in application logs
11. Default status for newly confirmed metadata: 'pending'

### 3. Autocomplete Metadata Enhancement

12. Metadata modal/form shall include autocomplete for:
    - **Authors**: Search existing authors from `authors` table, dropdown with suggestions
    - **Publishers**: Search existing publishers from `publishings` table
    - **Genres**: Search existing genres from `genres` table (Story 1.21)
    - **Themes/Categories**: Search existing themes from `theme_sets` table
13. Autocomplete shall support:
    - Partial text matching (case-insensitive)
    - Minimum 2 characters to trigger search
    - Keyboard navigation (arrow keys, enter to select, escape to close)
    - "Create new" option if no exact match found (admin only)
14. Creating new entries (Authors/Publishers/Genres/Themes) requires admin role
15. All autocomplete dropdowns shall be multilingual-aware (search across en/ru/he fields)

### 4. Metadata Modal UX Improvements

16. Enhanced metadata modal shall display:
    - Cover image preview (if uploaded in Story 1.21)
    - Publication file name and path (read-only)
    - All metadata fields from Story 1.21 with autocomplete
    - Confidence scores (from FileMetadata extraction) with color coding
    - Validation errors inline with field-specific messages
17. Modal shall support keyboard shortcuts:
    - `Ctrl+S` or `Cmd+S`: Save metadata
    - `Ctrl+Enter` or `Cmd+Enter`: Confirm and mark as published
    - `Escape`: Close without saving (with confirmation if dirty)
18. Modal shall have responsive design (mobile-friendly)

### 5. Publication Preview Page (`/publications/{id}/preview`)

19. New public route `/publications/{id}/preview` accessible to all users (guests + authenticated)
20. Preview page shall display:
    - Cover image (if available)
    - Title
    - Author(s) with links to author pages (if implemented)
    - Publication year
    - Publisher
    - Short description (max 500 chars or truncated `description` field)
    - Genre tags
    - Theme/Category badge
    - File format icon (PDF/EPUB/DOCX/etc.)
    - File size (human-readable: MB/KB)
21. Preview page for guests shall show:
    - "Register to access full content" CTA button
    - "Login" and "Sign Up" buttons prominently
    - Limited metadata (no download link, no full description)
22. Preview page for authenticated users shall show:
    - "View Full Publication" button (links to `/publications/{id}` detail page)
    - Download button (if publication status = 'published')
    - Engagement buttons: Like, Bookmark (from future stories)
23. Preview page shall be styled consistently with the rest of the application
24. Preview page shall support all three languages (EN/RU/HE with RTL)

### 6. Dashboard Filtering Integration

25. Dashboard shall reuse existing filter architecture from AdminDashboard:
    - Category filter (multiselect)
    - Author filter (autocomplete multiselect)
    - Date range filter
    - Genre filter (multiselect)
    - Text size filter (slider)
    - Alphabetical sort (A-Z/Z-A)
    - **Publication Status filter** (Published/Hidden/Pending - admin only)
26. Filters shall apply to both publication list AND metadata review queue
27. Filter state shall persist via URL parameters (shareable filtered URLs)
28. "Clear all filters" button shall reset to default view

### Autocomplete Implementation Details

**Autocomplete Requirements:**
[Source: User requirements + docs/architecture/tech-stack.md]
- Minimum 2 characters to trigger search
- Partial text matching (case-insensitive)
- Multilingual search (EN/RU/HE)
- Keyboard navigation (Arrow keys, Enter, Escape)
- "Create new" option for admins only
- Uses Alpine.js for client-side interaction
- Livewire for server-side search

**Search Query Pattern:**
```php
public function searchAuthors(string $query): array
{
    if (strlen($query) < 2) {
        return [];
    }

    return Author::query()
        ->where(function($q) use ($query) {
            $q->where('author', 'like', "%{$query}%")
              ->orWhere('author_low', 'like', "%".mb_strtolower($query)."%");
        })
        ->limit(10)
        ->get()
        ->map(fn($author) => [
            'id' => $author->id_author,
            'name' => $author->author
        ])
        ->toArray();
}
```

**Alpine.js Autocomplete Component Pattern:**
```html
<div x-data="{
    open: false,
    search: '',
    selected: null,
    results: [],
    async fetchResults() {
        if (this.search.length < 2) {
            this.results = [];
            return;
        }
        this.results = await $wire.searchAuthors(this.search);
        this.open = true;
    }
}" @click.away="open = false">
    <input
        type="text"
        x-model="search"
        @input.debounce.300ms="fetchResults()"
        @keydown.arrow-down.prevent="/* focus first result */"
        @keydown.arrow-up.prevent="/* focus prev result */"
        @keydown.enter.prevent="/* select focused */"
        @keydown.escape="open = false"
    />
    <div x-show="open" class="dropdown">
        <template x-for="result in results">
            <div @click="/* select result */">{{ result.name }}</div>
        </template>
        <div @click="/* create new */">Create new...</div>
    </div>
</div>
```
