# Story 1.21: Metadata Modal Enhancement and Publications Schema Cleanup

## Status
Draft

## Story

**As an** administrator,
**I want** an enhanced metadata modal with comprehensive publication fields and a simplified database schema,
**so that** I can efficiently manage all publication metadata in one place without database redundancy.

## Acceptance Criteria

1. The metadata review modal shall include the following fields with proper validation:
   - Title (required, max 255 chars)
   - Author(s) (dynamic list, min 1 author)
   - Publishing House/Publisher (optional, max 255 chars)
   - Publication Year (optional, 1000-current year)
   - Theme (optional, max 255 chars)
   - Genre(s) (dynamic list, max 255 chars each)
   - Content Type (dropdown: Book, Magazine, Article, Other, Custom types)
   - Cover Image (file upload, jpg/png/webp, max 5MB)
   - ISBN (optional, validated format)
   - DOI (optional, validated format)

2. The publications table schema shall be simplified by removing:
   - All confidence level tracking columns (`metadata_confidence_avg`, individual confidence scores)
   - Duplicate extracted metadata columns (`extracted_author_names`, `extracted_publication_year`, `extracted_publisher`, `extracted_isbn`, `extracted_doi`)
   - Metadata tracking columns (`metadata_source`, `metadata_confirmed_at`, `metadata_previous_values`)

3. The publications table shall retain ONLY the following metadata columns:
   - Core identity: `id_publication`, `title`, `title_low`, `upload_date`, `status`
   - Publication details: `issue_year`, `content_type_id`, `original_folder_path`, `word_count`
   - Legacy FK relationships: `id_publishing`, `id_part`, `id_issue_type`, `id_magazine`, `id_theme_set`, `id_author_set`
   - System fields: `_del_mark`, `add_int`, `add_char`, `created_at`, `updated_at`, `deleted_at`

4. Metadata shall be stored in the proper normalized locations:
   - Authors → `authors` table + `author_publication` pivot
   - Publisher → `publishings` table via `id_publishing` FK
   - Genres → `themes` table + `publication_theme` pivot OR new `genres` table
   - Theme/Category → `theme_sets` table via `id_theme_set` FK
   - Cover images → `files` table with type='cover' flag

5. The MetadataReviewForm component shall:
   - Display all new fields with appropriate UI controls (text inputs, dropdowns, file upload, dynamic lists)
   - Support adding/removing multiple authors and genres dynamically
   - Upload and preview cover images before saving
   - Validate all fields according to defined rules
   - Save confirmed metadata to the proper normalized database locations

6. The file_metadatas table shall remain unchanged as the intermediate extraction storage layer

7. Upon metadata confirmation, the system shall:
   - Create/update Author records in `authors` table
   - Create/link relationships in `author_publication` pivot table
   - Create/update Publisher record in `publishings` table if provided
   - Create/update Theme/Genre records and link via pivot tables
   - Upload cover image to storage and create File record
   - Update Publication record with all confirmed metadata references
   - Mark FileMetadata record as 'confirmed'

8. All existing metadata extraction functionality shall continue to work:
   - Extraction jobs still populate FileMetadata records
   - Admin can still review/edit extracted data
   - Confidence scores can still be displayed in the UI (from FileMetadata, not Publications)
   - Manual entry fallback remains available

9. A migration shall be created that:
   - Backs up existing extracted metadata to JSON in `metadata_previous_values` before deletion
   - Drops all confidence and extracted metadata columns from publications table
   - Does NOT drop data - attempts to migrate extracted data to proper normalized tables where possible
   - Provides rollback capability

### Current Database Structure

**Publications Table Current State:**
[Source: database/migrations/2025_10_30_201445_add_metadata_fields_to_publications_table.php]

Columns to REMOVE:
- `extracted_author_names` (JSON) - Duplicate of authors table
- `extracted_publication_year` (INT) - Duplicate of issue_year
- `extracted_publisher` (VARCHAR) - Duplicate of publishings table
- `extracted_isbn` (VARCHAR) - Should be in separate metadata table or custom fields
- `extracted_doi` (VARCHAR) - Should be in separate metadata table or custom fields
- `metadata_source` (ENUM) - Tracking not needed after confirmation
- `metadata_confidence_avg` (DECIMAL) - Confidence stays in FileMetadata
- `metadata_confirmed_at` (TIMESTAMP) - Tracked in FileMetadata
- `metadata_previous_values` (JSON) - Will be used for backup before dropping other columns

Columns to KEEP:
- `id_publication`, `title`, `title_low`, `upload_date`, `status`, `issue_year`
- `content_type_id`, `original_folder_path`, `word_count`
- `id_publishing`, `id_part`, `id_issue_type`, `id_magazine`, `id_theme_set`, `id_author_set`
- `_del_mark`, `add_int`, `add_char`, `created_at`, `updated_at`, `deleted_at`

**Normalized Data Storage Locations:**
[Source: docs/architecture/4-data-models-and-schema-changes.md]

- **Authors** → `authors` table + `author_publication` pivot (already exists)
- **Publisher** → `publishings` table via `id_publishing` FK (already exists)
- **Theme** → `theme_sets` + `themes` via `id_theme_set` FK (already exists)
- **Genres** → Either use `themes` table OR create new `genres` table + `genre_publication` pivot
- **Cover Images** → `files` table with new `file_type` column

**FileMetadata Table Structure:**
[Source: database/migrations/2025_10_26_224949_create_file_metadatas_table.php]

Keep unchanged - serves as extraction history:
- `extracted_data` (JSON) - Contains all extracted values with confidence scores
- `confidence_scores` (JSON) - Field-level confidence tracking
- `status` (ENUM: pending/processing/processed/confirmed/rejected/failed)
- This table remains the source of truth for extraction history and confidence

**Save Workflow Changes:**

OLD (Story 1.8):
```
confirmExtraction() → Update FileMetadata.status = 'confirmed'
                   → Store confirmed data in FileMetadata.extracted_data
                   → Fire MetadataConfirmed event
```

NEW (Story 1.21):
```
confirmExtraction() → DB Transaction START
                   → Create/Update Authors in authors table
                   → Create relationships in author_publication
                   → Create/Update Publishing if provided
                   → Link via publication.id_publishing
                   → Create/Update Genres in genres table
                   → Create relationships in genre_publication
                   → Upload cover image to storage/app/covers/
                   → Create File record with type='cover'
                   → Update Publication with all FKs
                   → Update FileMetadata.status = 'confirmed'
                   → DB Transaction COMMIT
                   → Fire MetadataConfirmed event
```

### Migration Strategy

**Step 1: Backup Existing Data**
```sql
UPDATE publications
SET metadata_previous_values = JSON_OBJECT(
  'extracted_author_names', extracted_author_names,
  'extracted_publication_year', extracted_publication_year,
  'extracted_publisher', extracted_publisher,
  'extracted_isbn', extracted_isbn,
  'extracted_doi', extracted_doi,
  'metadata_source', metadata_source,
  'metadata_confidence_avg', metadata_confidence_avg,
  'metadata_confirmed_at', metadata_confirmed_at
)
WHERE metadata_source IS NOT NULL;
```

**Step 2: Migrate Data to Normalized Tables** (Optional - can be manual)
- For each publication with extracted_author_names:
  - Create Author records if not exists
  - Create author_publication relationships
- For each with extracted_publisher:
  - Create Publishing record if not exists
  - Set id_publishing FK

**Step 3: Drop Columns**
```sql
ALTER TABLE publications
  DROP COLUMN extracted_author_names,
  DROP COLUMN extracted_publication_year,
  DROP COLUMN extracted_publisher,
  DROP COLUMN extracted_isbn,
  DROP COLUMN extracted_doi,
  DROP COLUMN metadata_source,
  DROP COLUMN metadata_confidence_avg,
  DROP COLUMN metadata_confirmed_at,
  DROP INDEX idx_publications_metadata_status;
```

**Rollback Strategy:**
```sql
-- Restore columns from metadata_previous_values JSON
ALTER TABLE publications ADD COLUMN extracted_author_names JSON NULL;
-- ... add other columns back
UPDATE publications SET extracted_author_names = JSON_EXTRACT(metadata_previous_values, '$.extracted_author_names');
```

### Security Considerations

**File Upload Security:**
[Source: docs/architecture/10-security-integration.md]
- Validate MIME types server-side (not just extension)
- Store files outside web root or in protected directory
- Generate unique filenames to prevent overwriting
- Scan uploaded images for malicious content (optional)
- Implement rate limiting on file uploads

**Data Validation:**
- All inputs validated via FormRequest or Livewire $rules
- ISBN format: `/^(?:ISBN(?:-1[03])?:?\s?)?(?=[0-9X]{10}$|(?=(?:[0-9]+[- ]){3})[- 0-9X]{13}$|97[89][0-9]{10}$|(?=(?:[0-9]+[- ]){4})[- 0-9]{17}$)(?:97[89][- ]?)?[0-9]{1,5}[- ]?[0-9]+[- ]?[0-9]+[- ]?[X0-9]$/`
- DOI format: `/^10\.\d{4,}\/\S+$/`
- XSS protection: Blade auto-escaping for all outputs

### Performance Considerations

**Transaction Boundaries:**
- Wrap entire save operation in DB transaction
- Rollback on any failure to maintain consistency
- Log errors before rolling back for debugging

**Query Optimization:**
- Use `firstOrCreate()` for Authors/Publishers to avoid duplicates
- Eager load relationships when displaying metadata
- Index foreign keys (already exists)
- Cache Content Types dropdown data (query once per request)

**File Storage:**
- Use Laravel's `Storage::putFileAs()` for atomic writes
- Store original filename in database for reference
- Generate thumbnail async (optional, future enhancement)
