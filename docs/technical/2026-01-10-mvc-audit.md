# Systems Analysis & MVC Consistency Audit
**Date:** 2026-01-10

## Executive Summary
The application is in a transitional state between legacy database patterns and modern Laravel architecture. While fully functional, this duality creates maintenance friction and potential fragility in data relationships.

---

## 1. MVC Architecture Analysis

### Controllers (The Good & The Bad)
- **Positive:** `DownloadController` demonstrates excellent modern practices, using Dependency Injection (DI) for testability.
- **Negative:** `FileViewController` is a "Fat Controller" (>800 lines) that violates the Single Responsibility Principle. It handles:
    - HTTP Request/Response (Correct)
    - File System Logic (Should be Service)
    - HTML Generation (Should be View)
    - Legacy Path Resolution (Should be Model/Helper)

### Views
- **Issue:** HTML content is being actively constructed inside PHP classes (`FileViewController`), bypassing the Blade templating engine. This makes frontend changes require backend code edits.

### Models
- **Inconsistency:** The application is split between two naming conventions for Primary Keys:
    - **Legacy:** `id_publication`, `id_part` (Requires explicit definition in every model).
    - **Modern:** `id` (Standard Laravel default).
- **Impact:** Developers must constantly context-switch or check definitions to write basic queries.

---

## 2. Data Hierarchy & Relationships

### The "File Metadata" Fragility
The `FileMetadata` model attempts to link to `Publication` and `File` models using a string parsing strategy rather than a Foreign Key.
- **Current Logic:** Parses `file_id` string `"123-filename.pdf"` to extract ID `123`.
- **Risk:** Extremely fragile. If a filename format changes, the relationship breaks silently.
- **Recommendation:** Add proper `publication_id` (integer) and `file_name` columns to the `file_metadatas` table.

### The "Part" Hierarchy
The hierarchy for multi-volume works (`Partition`) is implemented using legacy naming:
- `PartSet` (has many) -> `Part` (has many) -> `Publication`
- **Correction:** `PartSet` model defines `hasMany` to `Part`, but `Part` model does not have the corresponding `belongsTo` defined for `PartSet`. It only has `hasMany` to `Publication`.
- **Finding:** The hierarchy exists in the database but is incomplete in the Object Model (Inverse relationship missing).

---

## 3. Structural Contradictions

| Feature | Modern Approach | Legacy Approach | Location |
| :--- | :--- | :--- | :--- |
| **Storage Access** | Dependency Injection (`FileStorageService`) | Facade (`Storage::disk`) | `DownloadController` vs `FileViewController` |
| **Search Index** | Database Collation / Generated Columns | Redundant `_low` columns | `Publication`, `Author` |
| **Job Location** | `app/Jobs` | `app/Models` | `FolderScanJob` (is a Model, despite name) |

---

## 4. Recommendations Roadmap

### Phase 1: Stability (High Priority)
1.  **Refactor `FileViewController`:** Extract HTML generation into blade views.
2.  **Fix `FolderScanJob` Naming:** Rename to `ScanSession` to clarify it is a data entity, not a process.
3.  **Harden `FileMetadata`:** Create a migration to add real foreign keys.

### Phase 2: Modernization (Medium Priority)
4.  **Standardize DI:** Refactor all controllers to inject services instead of using Facades.
5.  **Unify Keys:** Long-term plan to migrate all `id_entity` primary keys to `id`.
